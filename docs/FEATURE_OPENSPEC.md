# Feature Plan — OpenSpec Integration

## Summary

Add a **final post-processing step** to the DockOck pipeline that takes
generated Gherkin output and feeds it into [OpenSpec](https://github.com/Fission-AI/OpenSpec)
to produce a full change set: **proposal**, **specs** (delta specs), **design**,
and **tasks** — all saved to disk inside an `openspec/` workspace.

## Can it be done?

**Yes — with caveats.**

OpenSpec is a Node.js CLI + AI-agent framework (npm package `@fission-ai/openspec`).
It is designed to run locally.  Its two complementary interfaces are:

| Interface | How it works | Relevant to DockOck? |
|-----------|--------------|----------------------|
| **CLI** (`openspec init`, `openspec status`, …) | Interactive & `--json` modes, runs as a child process | **Yes** — we shell out from Rust |
| **Slash commands** (`/opsx:propose`, `/opsx:ff`, …) | Interpreted by an AI coding assistant (Claude, Cursor, Copilot, …) inside the editor | **No** — these are prompt-level instructions for an AI agent chat, not callable APIs |

**Key insight:** OpenSpec does *not* expose a programmatic Rust/TypeScript library
API.  The slash commands (`/opsx:propose`, `/opsx:ff`, `/opsx:apply`) are *AI-agent
skills* — they instruct an AI assistant to read templates, call the CLI for status,
and write files.  There is no `openspec generate-from-gherkin` command.

### What we *can* do

DockOck already generates structured Gherkin.  We can:

1. **Scaffold** the OpenSpec folder structure ourselves (or via `openspec init --tools none`).
2. **Write the artifacts directly** using the Gherkin we already produce:
   - `proposal.md` — generated from a short LLM summarisation of the Gherkin feature.
   - `specs/<domain>/spec.md` — generated by converting our Gherkin scenarios into
     OpenSpec's *Requirement → Scenario* markdown format.
   - `design.md` — (optional) a placeholder or a short LLM-generated technical
     approach blurb.
   - `tasks.md` — generated by mapping each Scenario into a numbered task checklist.
3. **Validate** via `openspec validate --all --json` after writing.
4. **Optionally** call `openspec status --json` to confirm completeness.

This means DockOck becomes the *engine* that does what `/opsx:propose` normally
asks an AI agent to do, but we do it deterministically using our existing Gherkin
+ one extra LLM call (proposal summary) + Rust text transforms (specs, tasks).

### New module: `src/openspec.rs`

Responsibilities:

| Function | Purpose |
|----------|---------|
| `init_workspace(root)` | Ensure `openspec/` folder exists; optionally call `openspec init --tools none` if the CLI is installed |
| `export_change(name, gherkin_doc, summary_tx)` | Orchestrate writing all artifacts for one change |
| `gherkin_to_spec_md(gherkin_doc)` | Convert `GherkinDocument` scenarios → OpenSpec `spec.md` delta format |
| `gherkin_to_tasks_md(gherkin_doc)` | Convert scenarios → numbered task checklist |
| `generate_proposal(orchestrator, gherkin_text, status_tx)` | One LLM call to summarise the feature into intent/scope/approach |
| `write_design_stub(name)` | Write a minimal `design.md` (user can flesh out later) |
| `validate_change(name)` | If CLI present, run `openspec validate <name> --json` and parse output |

### Pipeline integration

In `app.rs` / `process_files()`:

```
Phase 1   — parse files  (existing)
Phase 1.5 — dispatch groups / singles  (existing)
Phase 2   — receive GherkinDocument results  (existing)
Phase 3   — NEW: if OpenSpec export enabled, run export_change() for each result
```

Phase 3 runs **after** all Gherkin results are collected (so cross-file context is
complete).  It can run concurrently per file/group through the same `status_tx`
channel.

## Gherkin → OpenSpec format mapping

### `specs/<domain>/spec.md`

```markdown
# <Feature Title> Specification

## Purpose
<Feature description from GherkinDocument>

## Requirements

### Requirement: <Scenario Title>
The system SHALL <derived from first Then step>.

#### Scenario: <Scenario Title>
- GIVEN <given steps>
- WHEN <when steps>
- THEN <then steps>
- AND <and steps>

... (one Requirement block per Scenario)
```

### `tasks.md`

```markdown
# Tasks

## 1. <Scenario 1 title>
- [ ] 1.1 Implement: <Given context setup>
- [ ] 1.2 Implement: <When action>
- [ ] 1.3 Verify: <Then expected outcome>

## 2. <Scenario 2 title>
- [ ] 2.1 …
```

### `proposal.md`

Generated via one LLM call with a preamble like:

```
You are a technical writer. Given the following Gherkin Feature file,
produce a concise OpenSpec proposal with sections:
- Intent (1-2 sentences: what problem does this solve?)
- Scope (bullet list: in-scope / out-of-scope)
- Approach (1 paragraph: high-level technical approach)
Do not output Gherkin. Do not add extra sections. Be concise.
```

### `design.md` (stub)

```markdown
# Design: <Feature Title>

## Technical Approach
_To be completed._

## File Changes
_To be completed after implementation planning._
```

## OpenSpec CLI detection

DockOck should work **with or without** the OpenSpec CLI installed:

| CLI present? | Behaviour |
|--------------|-----------|
| Yes | `openspec init`, `openspec validate --json` available; use them |
| No | DockOck writes the folder structure and files directly (pure Rust); skip validation |

Detection: `which openspec` / `where openspec` or `std::process::Command::new("openspec").arg("--version")`.

## Implementation phases

### Phase 1 — Core export (Rust-only, no CLI dependency)

1. Create `src/openspec.rs` with `gherkin_to_spec_md`, `gherkin_to_tasks_md`,
   `write_design_stub`, and `export_change`.
2. Add `openspec_enabled: bool` + `openspec_output_dir: String` to `DockOckApp`.
3. After all results are collected, write `openspec/changes/<name>/` with
   `proposal.md` (via LLM), `specs/…/spec.md`, `tasks.md`, `design.md`.
4. UI: checkbox in settings, "Export to OpenSpec" button, status indicators.

### Phase 2 — LLM proposal generation

5. Add `PROPOSAL_PREAMBLE` to `llm/mod.rs`.
6. Add `generate_proposal()` method on `AgentOrchestrator`.
7. Wire into `export_change` flow.

### Phase 3 — CLI integration (optional)

8. Detect `openspec` CLI on startup.
9. Run `openspec init --tools none` if workspace doesn't exist yet.
10. Run `openspec validate --all --json` after export and surface results in UI.

### Phase 4 — Polish

11. Per-file and per-group export alongside "Save" buttons.
12. "Export All" button.
13. Show validation warnings/errors in right panel.
14. Remember export settings across sessions (serde persistence).

## Dependencies

No new Rust crate dependencies required.  File I/O uses `std::fs`.
CLI invocation uses `std::process::Command`.
The OpenSpec npm package (`@fission-ai/openspec`) is **optional** at runtime.

## Risks & mitigations

| Risk | Mitigation |
|------|------------|
| OpenSpec format changes between versions | Pin to known format; validate with CLI when available |
| Proposal LLM call adds latency | Make it optional (checkbox); run in parallel with other exports |
| User expects full `/opsx:apply` behaviour | Clearly document that DockOck generates *planning* artifacts only; implementation is a separate step |
| Large projects produce many change folders | Group-based Gherkin → single change folder per group |

## Open questions

1. **Naming convention for changes:** Use file stem (`D028`) or a slugified feature
   title (`user-authentication`)?  → Suggest feature title extracted from Gherkin,
   falling back to file stem.
2. **Delta vs. full specs:** Should we always generate delta format (`## ADDED Requirements`)
   or full format?  → Default to full for new projects; delta when existing specs detected.
3. **Multiple files → one change vs. many changes:** Each file/group gets its own
   change folder, or all results merge into a single change?  → One change per
   file/group, matching current 1:1 Gherkin output model.
