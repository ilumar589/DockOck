Delivery Loop

This visual is meant for stakeholder explanation. It shows why the process is reliable: approved artifacts first, explicit OpenSpec change definition next, bounded OpenCode execution after that, then review, repair, and maintenance loops only where they belong.

Executive summary Workflow map Agent routing

Focus modes

Highlight the straight-through happy path, the repair loop, or the maintenance loop that tightens prompts after repeated workflow drift.

1

Approved Artifacts

Corrected Gherkins, Markdown knowledge, graph outputs, and indexed documentation form the approved starting point.

2

OpenSpec Change

Proposal, spec, design, and tasks make the change explicit enough to review before implementation starts.

3

OpenCode Execution

The selected agent stack executes the feature in bounded slices rather than one broad generation pass.

4

Review and Validation

Code review, QA, security, performance, and generated tests decide whether the slice passes, needs repair, or exposes recurring workflow mistakes.

5

Accepted Slice

The change is ready for merge, release hardening, or the next bounded slice.
